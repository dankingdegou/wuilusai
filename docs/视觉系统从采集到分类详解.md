# 比赛视觉系统：从相机采集到三箱豆类分类

本文说明当前已经在 PC 上跑通的第一版视觉系统。它的目标是：相机从固定俯视位置拍摄三个箱子，判断每个箱子里的豆类是绿豆、黄豆还是白豆，并给后续机械臂提供箱子编号、类别和置信度。

当前系统不是 YOLO，也没有把整张图直接丢给神经网络。它利用了比赛现场的固定条件：相机固定、箱子位置固定、类别固定且只有三类。这种做法更容易解释、调试和移植到 Jetson。

## 1. 最终输入与输出

输入是一张 Orbbec RGB 相机图片，当前统一使用：

```text
MJPG，1280 × 960，30 FPS
```

输出是 JSON。每个箱子都包含固定 ROI、颜色统计特征、预测类别和置信度。例如：

```json
{
  "boxes": {
    "left": {
      "decision": {"label": "mung_bean", "confidence": 0.979}
    },
    "center": {
      "decision": {"label": "soybean", "confidence": 0.951}
    },
    "right": {
      "decision": {"label": "white_bean", "confidence": 0.964}
    }
  }
}
```

类别名含义如下：

| 程序类别名 | 含义 |
| --- | --- |
| `mung_bean` | 绿豆 |
| `soybean` | 黄豆 |
| `white_bean` | 白豆 |

后续机械臂节点不需要理解图像，只需要读取 `left / center / right` 的分类结果，再决定去哪个箱子抓取。

## 2. 当前目录结构

核心文件都在 `vision/`：

```text
vision/
├── capture_vision_rgb.py       # 交互式拍照，默认 1280×960
├── calibrate_rois.py           # 一次性框选三个箱子的 ROI
├── config/boxes.yaml           # 相机、ROI、原型等实际配置
├── data/calibration/           # 手动采集的训练照片
├── data/labels.yaml            # 每张图片的人工类别标注
├── label_dataset_groups.py     # 按连续照片组快速标注
├── feature_model.py             # 训练和推理共用的特征提取
├── train_bean_classifier.py    # 训练 SVM 并做严格验证
├── models/bean_classifier.joblib # 训练得到的模型
├── run_box_vision.py           # 单帧视觉推理主程序
└── run_vision_pc.sh            # PC 上的推荐推理入口
```

项目根目录的 `capture_orbbec_rgb.py` 是通用相机拍照程序。`capture_vision_rgb.py` 复用了它的交互能力，只是把默认分辨率改为视觉管线需要的 `1280×960`，并把照片放入 `vision/data/calibration/`。

## 3. 为什么统一使用 1280×960

ROI 本质上是像素坐标，例如当前配置中的：

```yaml
left:
  x: 31
  y: 530
  w: 226
  h: 100
```

它只在相同的相机姿态和相同分辨率下有意义。如果用 `2048×1536` 拍照、却用 `1280×960` 的 ROI 推理，框选区域会错位，分类必然失效。

因此应遵守：

1. 标定照片使用 `1280×960`；
2. 训练图片使用 `1280×960`；
3. 日常推理也使用 `1280×960`；
4. 相机支架、焦距、拍摄方向固定后，不能随意移动。

文件名中旧的 `2048x1536` 字样只是此前通用脚本的命名残留；实际检查过，当前训练图片的真实像素均为 `1280×960`。

## 4. 整个工作流

```text
固定 Orbbec 相机
        ↓
拍摄多组不同摆放位置的照片
        ↓
在一张代表图上框选 left / center / right ROI
        ↓
给每组照片标注三箱的真实豆类
        ↓
从每个 ROI 提取颜色分布特征
        ↓
训练 SVM 分类器
        ↓
按“整组照片留出”验证
        ↓
实时拍一帧 → 三箱分类 JSON + 调试图
```

其中只有两步需要人工判断：ROI 框选和训练照片类别标注。其余步骤由程序自动完成。

## 5. 第一步：相机采集

交互式采集命令：

```bash
cd ~/ros2_ws/wuliusai/vision
python3 capture_vision_rgb.py
```

行为与通用拍照脚本一致：

- Space 或 Enter：拍照；
- Q 或 Esc：退出；
- 相机会先预热；
- 每次连读多帧，选择清晰度最高的一帧保存；
- 默认保存到 `data/calibration/`。

采集时应让三个箱子均完整出现在画面中，并让豆子分别放在箱内多个位置。训练数据不应只拍一种摆放方式，否则模型会把“豆子位置”误认为“豆子类别”。

## 6. 第二步：ROI 标定

ROI 是每个箱子中真正参与分类的矩形区域。框选时要避开透明箱壁、橙色垫箱、黑色桌面和背景。

```bash
cd ~/ros2_ws/wuliusai/vision
python3 calibrate_rois.py \
  --config config/boxes.yaml \
  --image data/calibration/<一张三箱都有豆的照片>.jpg
```

程序依次要求框选 `left`、`center`、`right`。每次鼠标拖动后按 Enter 或 Space 确认。结果写入 `config/boxes.yaml`。

ROI 的意义是把整张 1280×960 图片缩小成三个小问题：

```text
整张图 → 左箱豆粒区域
       → 中箱豆粒区域
       → 右箱豆粒区域
```

这样窗帘、地面、箱体边框都不会参与分类，稳定性会显著提高。

## 7. 第三步：人工标注为什么按“组”做

一次摆放完成后通常连续拍了多张相似图片。逐张标注 52 张没有必要，因此 `label_dataset_groups.py` 以时间间隔把照片分成 10 组，一组只标注一次，自动套用到该组全部图片。

```bash
python3 label_dataset_groups.py
```

终端输入顺序固定为：

```text
left center right
```

例如：

```text
1 2 3
```

表示左箱绿豆、中箱黄豆、右箱白豆。标注结果写入 `data/labels.yaml`。当前数据集已经完成标注：52 张图，每类豆子各有 52 个 ROI，共 156 个训练样本。

## 8. 特征：程序实际从图像中学到了什么

单纯读取一个像素的 RGB 值不可靠。阴影、高光和相机自动曝光都会改变它。

`feature_model.py` 对每个 ROI 提取以下信息：

1. Lab 三个通道的 5%、15%、25%、50%、75%、85%、95% 分位数、均值与标准差；
2. HSV 三个通道的同类统计；
3. Lab 的 `a/b` 色度二维直方图。

Lab 中：

- `L` 主要表示亮度；
- `a` 表示绿到红方向；
- `b` 表示蓝到黄方向。

HSV 中：

- `H` 表示色相；
- `S` 表示饱和度；
- `V` 表示亮度。

分位数与直方图描述的是“这个箱子里的整体颜色分布”，比一个中位数更能区分浅色白豆、黄色黄豆和深色绿豆，也更能容忍阴影和少量背景。

## 9. 为什么没有继续使用单一 Lab 原型

最初程序用每类一个 Lab 中位数和距离阈值做判断。这很直观，但在当前照片上的训练集准确率只有 **49.36%**。

根本原因是：同一种豆子在不同箱位、不同阴影下亮度变化很大；一个固定颜色点不能代表所有情况。

因此现在保留 Lab 原型功能作为简单备用，而正式推理使用 SVM 分类器。

## 10. SVM 训练与验证

训练命令：

```bash
cd ~/ros2_ws/wuliusai/vision
./.venv/bin/python train_bean_classifier.py
```

`vision/.venv` 是视觉模块独立的 Python 环境，里面包含 `scikit-learn`。这样不会修改 ROS2 或系统 Python 的依赖。

训练器是：

```text
StandardScaler + RBF SVM
```

`StandardScaler` 先把不同数值范围的特征缩放到相近尺度；RBF SVM 再学习三类豆子在特征空间中的非线性边界。

验证不能把同一摆放组的相邻图片同时放进训练集和测试集，否则结果会虚高。当前采用的是 `LeaveOneGroupOut`：

```text
每次拿一整组“新摆放位置”的照片做测试，
其余九组用来训练；
循环直到十组都测试过一次。
```

当前结果：

```text
样本数：156
摆放组数：10
正确数：149
严格验证准确率：95.51%
```

报告保存在：

```text
vision/output/cross_validation.json
```

这说明模型不是只记住了同一组连拍图，而是在未参与训练的摆放组上也能工作。它仍不是最终比赛成绩；比赛前还应该单独拍一批从未用于训练的新图片做最终验收。

## 11. 日常 PC 推理

实时相机推理：

```bash
cd ~/ros2_ws/wuliusai/vision
./run_vision_pc.sh
```

离线检查某张图片：

```bash
./run_vision_pc.sh \
  --image data/calibration/orbbec_20260715_140453_006_2048x1536.jpg
```

每次推理会生成：

```text
output/
├── scene_<时间>.jpg     # 实际用于判断的原图
├── debug_<时间>.jpg     # 画有 ROI、类别、置信度的调试图
└── result_<时间>.json   # 提供给上层程序的结构化结果
```

调试图是现场排错的关键。若分类错误，优先检查：ROI 有没有偏移、相机有没有移动、箱内是否混入其他豆类、图片是否过曝或相机是否断开。

## 12. 与机械臂的接口边界

当前视觉模块只负责“识别哪个箱子是什么”。机械臂控制模块应读取 JSON，并按照预先标定的箱子坐标移动。

建议上层接口先定义为：

```yaml
box_id: left
bean_type: mung_bean
confidence: 0.98
```

后续加入单颗豆子分割后，再增加：

```yaml
grasp_pixel:
  x: 123
  y: 456
grasp_score: 0.91
```

这样视觉负责给出“哪里可以抓、是什么”，运动控制负责“如何安全到达并抓取”。二者不要互相直接操作对方的硬件。

## 13. 何时需要重新标定或重新训练

以下情况必须重新进行 ROI 标定，并建议重新采样训练：

- 相机高度、俯仰角或朝向改变；
- 分辨率从 1280×960 改变；
- 三个箱子的固定位置改变；
- 更换相机；
- 背景、箱体或主要光照条件发生很大变化。

仅仅是豆子在箱内换位置，不需要重新标定；这正是当前训练集已经覆盖的情况。

## 14. 移植到 Jetson 的顺序

在 PC 上完成并验收后，再移植：

1. 复制 `vision/` 下的脚本、`config/boxes.yaml`、`models/bean_classifier.joblib`；
2. Jetson 使用同样的固定 Orbbec 路径和同样的 1280×960 参数；
3. 建立 Jetson 专用 Python 虚拟环境，安装 `scikit-learn`，但复用系统 OpenCV；
4. 在 Jetson 上离线运行一张 PC 已验证照片，结果应一致；
5. Jetson 连接相机后再运行实时推理；
6. 最后才将 JSON 输出接入机械臂动作逻辑。

移植的原则是：先验证“同一张图片得到同一结果”，再验证“同一现场实时画面得到合理结果”，最后才允许机械臂根据视觉结果动作。

## 15. 下一阶段

当前版本完成的是“箱子级分类”。下一阶段是“单颗豆子定位”：

```text
箱子 ROI
  → 豆粒前景分割
  → 粘连豆子的分离
  → 每颗豆子的轮廓、中心点、朝向
  → 选择最孤立、最容易抓的一颗
```

在完成单颗定位前，机械臂应只把当前视觉结果用于确认箱子类别，不能假设它已经知道每一颗豆子的准确抓取位置。
