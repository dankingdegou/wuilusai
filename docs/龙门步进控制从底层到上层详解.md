# 龙门步进控制：从 STM32 底层到 ROS2 上层的设计与流程

这份文档解释比赛龙门架的两台步进电机如何从电脑命令一直传递到 STEP/DIR 脉冲，以及为什么不能把两侧长滑轨当成两个完全独立的电机来控制。

当前工程由两部分组成：

```text
/home/rog/ros2_ws/bjdj
    └─ STM32F103VET6 固件：实时生成两路 STEP/DIR 脉冲

/home/rog/ros2_ws/wuliusai/ros2_ws
    └─ ROS2 包：物理单位换算、串口协议、龙门同步 Action、状态与安全保护
```

---

## 1. 全链路总览

```text
比赛状态机 / 视觉程序 / 终端命令
              │ ROS2 Action / Service
              ▼
wuliusai_stepper_controller 节点
  - mm / degree → STEP 脉冲数
  - 软限位、人工设零、超时、取消
  - 同步两侧龙门滑轨
              │ 115200 UART 帧
              ▼
Type-C → CH340（USB 转串口）→ STM32 USART1（PA9/PA10）
              │
              ▼
STM32F103VET6
  - TIM2：YL 左轨 STEP/DIR，PA0/PA1
  - TIM3：YR 右轨 STEP/DIR，PA2/PA3
              │
              ▼
两路步进驱动器 → 两侧长滑轨 → 同一个龙门架
```

电脑从 Type-C 口看到的是 CH340（`1a86:7523`），不是 STM32 原生 USB。CH340 把 USB 数据转换成普通串口数据，再由 STM32 的 USART1 接收。因此串口波特率固定为 `115200`。

## 2. 为什么龙门架必须有“同步轴”

龙门架的两台电机分别在左右两条长滑轨上，但它们共同拖动同一个刚性横梁。

```text
左电机 ── 左滑轨 ──┐
                   ├── 龙门横梁
右电机 ── 右滑轨 ──┘
```

如果上层分别发送两条独立命令：

```text
move_axis(左侧)
move_axis(右侧)
```

就会有几个风险：

1. 两条串口命令到达时间不同，启动时间不同。
2. 左右速度、加速度或脉冲当量不一致，运动结束时间不同。
3. 一侧堵转、掉步或被卡住时，另一侧仍在推，龙门架会跑斜甚至卡死。
4. 比赛状态机若误操作一侧，机械风险很大。

因此正式调用使用一个语义更高的动作：

```text
move_gantry(target=120 mm)
```

它的意思不是“让两个普通电机各动一次”，而是“让整个龙门架移动到共同位置”。上层会针对左右不同的脉冲当量计算各自脉冲数，再通过一条 STM32 同步命令下发。

## 3. STM32 底层：哪些文件在做什么

| 文件 | 作用 |
|---|---|
| `bjdj/Core/Src/main.c` | 串口协议解析、运动状态机、速度曲线、同步命令处理 |
| `bjdj/Core/Src/tim.c` | TIM2、TIM3 初始化；定时器中断转交给步进控制逻辑 |
| `bjdj/Core/Src/stm32f1xx_it.c` | TIM2、TIM3、USART1 的硬件中断入口 |
| `bjdj/Core/Inc/main.h` | 两轴 `StepperAxis` 运行状态结构 |
| `bjdj/bjdj.ioc` | CubeMX 引脚、时钟、中断优先级配置 |

### 3.1 两路引脚

```text
axis 0 / YL（左轨）
  PA0 → YL_STEP
  PA1 → YL_DIR
  TIM2 负责定时

axis 1 / YR（右轨）
  PA2 → YR_STEP
  PA3 → YR_DIR
  TIM3 负责定时
```

STEP 脉冲的每一个上升沿通常对应驱动器的一个微步/步进脉冲；实际是多少机械位移，由电机步距角、驱动器细分和传动机构共同决定。

### 3.2 为什么使用两个定时器

系统时钟为 72 MHz。TIM2/TIM3 的预分频器设为 71，因此定时器每 `1 µs` 计数一次。

若目标速度为 `500 pps`（pulses per second，每秒脉冲数），单个完整周期约为：

```text
1,000,000 / 500 = 2000 µs
```

固件把高电平、低电平各占一半，定时器重装值约为：

```text
1,000,000 / (2 × 500) - 1 = 999
```

分别使用 TIM2 和 TIM3 后，左右两侧可独立产生不同频率的脉冲；同步龙门命令则在同一条 UART 帧中配置并启动两侧。

### 3.3 每个轴的运行状态

`StepperAxis` 保存每个电机当前这一次运动的关键数据：

```text
target_steps         目标脉冲数
step_count           已输出脉冲数
target_pps           目标速度
current_pps          当前速度
acceleration_pps2    加速度
request_id           本次命令编号
pulse_is_high        当前 STEP 是否处于高电平
running              是否正在运动
done_pending         是否刚完成、等待主循环回报
```

定时器中断只做时间敏感的工作：翻转 STEP、计数、更新速度、发现完成后停止定时器。串口发送放在主循环中完成，避免在中断里等待串口导致脉冲抖动。

### 3.4 加减速为什么必要

步进电机不能无限快地从静止跳到高速；负载较大时会失步。固件使用近似梯形速度曲线：

```text
起步速度 → 加速 → 目标速度 → 提前减速 → 停止
```

上层传给固件的参数是：

```text
max_pps             最高脉冲速度
acceleration_pps2   加速度
```

两者都有固件保护范围：速度 `100–5000 pps`，加速度 `100–20000 pps²`。

## 4. STM32 串口协议

每帧格式如下：

```text
AA 55 | command | length | payload | CRC8
```

- 帧头：`AA 55`
- CRC8：覆盖 `command + length + payload`
- CRC 多项式：`0x07`
- 当前固件协议版本：`1.1`

### 4.1 主要命令

| 命令 | 名称 | 作用 |
|---:|---|---|
| `0x10` | INFO | 查询协议版本和轴数量 |
| `0x11` | MOVE_AXIS | 单轴相对脉冲运动，仅调试用途 |
| `0x12` | STOP_AXIS | 停止单轴，或 `0xFF` 停止全部 |
| `0x13` | GET_STATE | 查询单轴或两轴状态 |
| `0x14` | MOVE_SYNC | 双轴同步运动，龙门正式使用 |
| `0x80` | INFO_REPLY | INFO 的回包 |
| `0x81` | STATUS | 接受、完成、停止、错误等状态回包 |

### 4.2 单轴移动 `0x11`

```text
axis: uint8
steps: int32          正负决定 DIR
max_pps: uint32
acceleration_pps2: uint32
request_id: uint16
```

### 4.3 龙门同步移动 `0x14`

```text
steps_axis_0: int32
steps_axis_1: int32
max_pps: uint32
acceleration_pps2: uint32
request_id: uint16
```

固件找出两侧中脉冲数较多的一侧，将它作为时间基准；另一侧的速度和加速度按脉冲数比例缩放。这样即使左右脉冲当量不同，也会尽量同时完成。

例如：

```text
左侧需要 2000 脉冲
右侧需要 1800 脉冲
最大速度设为 1000 pps

左侧速度 = 1000 pps
右侧速度 = 900 pps
```

两轴仍各由独立定时器输出脉冲，但它们由同一帧命令启动，启动间隔仅为固件连续配置的微秒级时间，而不是两条上位机串口命令之间的不确定延迟。

### 4.4 状态回包

状态包包括：轴号、状态码、请求编号、已执行步数。

| 状态码 | 意义 |
|---:|---|
| `0x00` | 已接受命令 |
| `0x01` | 正常完成 |
| `0x02` | 参数或执行错误 |
| `0x03` | 已停止 |
| `0x04` | 该轴正在忙 |
| `0x10` | 上电就绪 |
| `0x20` | 查询到的当前运行状态 |

`request_id` 很重要：上位机可区分“本次命令的完成回包”和“上一次残留的回包”。

## 5. Python 串口层

路径：

```text
ros2_ws/src/wuliusai_stepper_controller/
  wuliusai_stepper_controller/protocol.py
```

`StepperSerial` 是 ROS2 节点和底层测试脚本共用的串口封装。它负责：

1. 打开 `/dev/stepper_controller`；没有别名时测试脚本会回退到 `/dev/ttyUSB0`。
2. 后台线程持续读取 CH340 串口数据。
3. `FrameParser` 从连续字节流恢复完整帧，并丢弃噪声、错误 CRC 或不完整帧。
4. 根据 `request_id` 等待正确的 ACK、DONE、STOPPED 或 ERROR 回包。
5. 在 Ctrl-C、超时、取消等异常路径下发送 STOP。

这层的价值是把“字节流 + CRC + 并发回包”变成 Python 方法，例如：

```python
board.move(axis=0, steps=100, max_pps=200, acceleration_pps2=500, request_id=101)
board.move_sync(steps_axis_0=100, steps_axis_1=-100,
                max_pps=200, acceleration_pps2=500, request_id=201)
board.stop(0xFF)
```

上层正常不直接调用这些方法，而是通过 ROS2 Action 调用；它们主要用于通信与硬件调试脚本。

## 6. ROS2 上层封装

ROS2 工作区中有两个包：

```text
wuliusai_stepper_msgs
  ├─ MoveAxis.action          独立轴调试接口
  ├─ MoveGantry.action        龙门正式接口
  ├─ SetAxisZero.srv
  ├─ SetGantryZero.srv
  └─ StopAxis.srv

wuliusai_stepper_controller
  ├─ node.py                  ROS2 节点和安全逻辑
  ├─ protocol.py              串口协议层
  ├─ config/stepper_controller.yaml
  └─ launch/stepper_controller.launch.py
```

### 6.1 为什么移动使用 Action，而不是普通 Service

一次长轨移动可能持续数秒。它需要：

- 目标是否被接受；
- 运行过程中的反馈；
- 完成/失败结果；
- 取消能力；
- 超时处理。

ROS2 Action 正好提供这些能力；普通 Service 更适合“立即完成”的操作，例如设零、停止。

### 6.2 `MoveGantry.action`

```text
Goal
  relative: bool        true=相对移动，false=绝对移动
  target: float64       mm 或 deg，由 YAML unit 决定
  max_speed: float64    物理单位每秒；0 表示使用 YAML 默认值

Result
  success: bool
  message: string
  final_position: float64

Feedback
  left_position: float64
  right_position: float64
  current_speed: float64
  state: string
```

调用时，上层先读取左右两侧配置：

```text
pulses_per_unit
direction
max_speed
acceleration
min_position / max_position
```

然后把共同目标换算为两侧各自的原始脉冲：

```text
steps = round((target - current_position) × pulses_per_unit × direction)
```

例如龙门目标是 `120 mm`，两侧的脉冲当量分别是 `40.0`、`42.0 pulses/mm`：

```text
axis 0: 120 × 40 = 4800 pulses
axis 1: 120 × 42 = 5040 pulses
```

之后调用 `MOVE_SYNC`，STM32 负责按比例安排两侧速度。

### 6.3 人工设零与绝对位置

当前没有原点开关或编码器。STM32 只能知道“自己发出了多少脉冲”，不知道机械是否真的移动了对应距离。

因此上层区分两种状态：

```text
referenced = false
  刚上电、重连、停止异常、可能失步之后。
  允许谨慎的相对移动；拒绝绝对移动。

referenced = true
  用户确认龙门在机械零点后调用 set_gantry_zero。
  此后允许带软限位的绝对移动。
```

`set_gantry_zero(position=0.0)` 会一次把左右两侧的理论位置都设成 `0.0`，而不会只设置其中一侧。

### 6.4 软限位

YAML 中应填写机械真实行程：

```yaml
axis_0.min_position: 0.0
axis_0.max_position: 250.0
axis_1.min_position: 0.0
axis_1.max_position: 250.0
```

上层会在发送命令前检查目标是否在范围内。软限位可以防止程序计算错误，但不能代替物理限位开关：如果步骤标定错误或发生失步，软件仍可能误判实际位置。

## 7. 推荐调试与上线流程

### 阶段 A：通信

```bash
cd /home/rog/ros2_ws/wuliusai
python3 scripts/test_stepper_communication.py
```

预期：`STM32 info: (1, 1, 2)`。此脚本不驱动电机。

### 阶段 B：单侧方向

```bash
python3 scripts/test_single_stepper.py
```

先用 `100` 脉冲确认 axis 0、axis 1 各自正负方向。此时不要让两侧在有刚性约束的情况下做大幅不同步运动。

### 阶段 C：双侧同步小位移

```bash
python3 scripts/test_gantry_sync.py
```

根据阶段 B 的结果填写左右原始脉冲符号。例如两侧电机安装方向相反时，可能需要左侧 `100`、右侧 `-100` 才能让龙门同向。

### 阶段 D：标定

发送已知脉冲数 `N`，测量实际位移 `D`：

```text
pulses_per_unit = N / D
```

分别标定两侧，将结果写入 `stepper_controller.yaml`。不要假设两边完全相同；皮带轮、丝杠、细分设置或机械安装差异都可能导致不同。

### 阶段 E：ROS2 龙门动作

```bash
cd /home/rog/ros2_ws/wuliusai/ros2_ws
source /opt/ros/jazzy/setup.bash
source install/setup.bash
ros2 launch wuliusai_stepper_controller stepper_controller.launch.py
```

另开终端：

```bash
# 先整体设零
ros2 service call /stepper_controller/set_gantry_zero \
  wuliusai_stepper_msgs/srv/SetGantryZero "{position: 0.0}"

# 再小幅同步移动
ros2 action send_goal /stepper_controller/move_gantry \
  wuliusai_stepper_msgs/action/MoveGantry \
  "{relative: true, target: 5.0, max_speed: 0.0}" --feedback
```

确认方向、位置、同步性都正确后，再逐步提高行程和速度。

## 8. 当前系统能保证什么，不能保证什么

### 已经具备

- 两路独立 STEP/DIR 定时器输出。
- 单帧双轴同步启动与按脉冲比例的速度缩放。
- 加减速、停止、完成和错误回包。
- ROS2 Action 的结果、反馈、取消与超时保护。
- 单位换算、理论位置、软限位、整体人工设零。
- 龙门模式下禁止单侧正式调用。

### 仍无法自动解决

- 一侧电机实际堵转或丢步。
- 两侧滑轨机械阻力不同导致的真实位置偏差。
- 断电后的真实位置恢复。
- 撞到物理端点时的可靠保护。

原因是目前没有原点开关、限位开关、编码器或电机驱动器报警输入。软件只能验证“命令是否发送、脉冲是否输出”，不能测量“龙门是否真的在正确位置”。

比赛前最值得增加的硬件是：

1. 至少一个共同原点开关；更理想的是左右各一个原点开关。
2. 两端限位开关，直接接入 STM32 并在中断中停止对应轴。
3. 若预算允许，使用带报警输出或闭环反馈的驱动器。

加入原点开关后，可以新增 `home_gantry` Action：低速向零点移动，两个开关均触发后停止，再自动调用整体设零。这会把当前“人工确认”变成可重复的比赛启动流程。
